## Why

Today's OSS layout (`Drama/{slug}/{ep_dir}/{ladder}/...`) is shared between this server (staging / 编辑端) and whatever business server eventually hosts SDK traffic (prod). The "manual sync" UX promised in `business-server-sync` (step 6) — operator reviews on staging, then explicitly clicks "同步" to publish — is **architecturally impossible with this layout**: when an operator re-uploads a video on staging, the OSS path is overwritten and prod's SDK clients immediately serve the un-reviewed video. Metadata sync gating buys nothing if the underlying media files leak through.

This change splits OSS into two namespaces under the same bucket: `Drama/staging/{slug}/...` (where the encoder publishes) and `Drama/prod/{slug}/...` (what the business server's SDK traffic reads). It also introduces the primitives that step 6 will call when the operator clicks sync — OSS server-side copy (`staging → prod`), prod-flavored m3u8 generation, and prod-side delete. With this change in place, step 6's sync handler is "call `publish_to_prod`, then POST metadata," and editing on staging is fully isolated from prod.

This is a hard prerequisite for `business-server-sync`. The system has no production data, so the path migration is destructive (delete `OUT_DIR`, delete OSS bucket contents, restart) — the same posture taken in earlier changes.

## What Changes

- **BREAKING** OSS upload paths change from `Drama/{slug}/{ep_dir}/{ladder}/...` to `Drama/staging/{slug}/{ep_dir}/{ladder}/...`. The pipeline-finish hook (worker → `publish_ladder`) writes only to staging.
- Local m3u8 files generated by `publish_ladder` reference the staging absolute URL: `https://{bucket}.{endpoint}/Drama/staging/{slug}/{ep_dir}/{ladder}/...`. `#EXT-X-KEY:URI` continues to be the relative `/drm/{slug}/ep-{n}/key`.
- `EpisodeInfo.initUrl` / `firstSegUrl` returned by **this** server (staging-side `/api/episodes/...`) reference the staging absolute URL.
- Add new public constants in `app/oss_upload.py`: `oss_staging_public_base_url` and `oss_prod_public_base_url`. The existing `oss_public_base_url` remains the bucket-level base (`https://photobundle.../Drama`) for internal composition; callers MUST use the env-specific child constant for URL construction.
- Add new functions in `app/publish.py`:
  - `publish_ladder_to_prod(slug, ep_dir, ladder) -> str`: OSS server-side copies `Drama/staging/{slug}/{ep_dir}/{ladder}/*` → `Drama/prod/{slug}/{ep_dir}/{ladder}/*`, reads the local staging-flavored m3u8, and returns a **prod-flavored** m3u8 text (init/segment URLs swapped to prod prefix; `#EXT-X-KEY:URI` left intact). Does **NOT** write the prod m3u8 to disk; the caller (step 6's sync handler) ships it over to the business server.
  - `unpublish_ladder_from_prod(slug, ep_dir, ladder) -> None`: deletes prod objects under that ladder dir. Idempotent.
  - `unpublish_drama_from_prod(slug) -> None`: deletes everything under `Drama/prod/{slug}/`. For drama-level delete sync.
- Add OSS primitives in `app/oss_upload.py`:
  - `copy_object(src_key, dst_key)`: OSS server-side copy within the same bucket.
  - `list_with_prefix(prefix) -> list[str]`: paginated listing of all keys under a prefix.
  - `batch_delete(keys: list[str]) -> None`: bulk delete (oss2 supports up to 1000 keys per call; helper handles chunking).
- Wire staging-side OSS cleanup into existing delete handlers: when `DELETE /admin/episodes/{slug}/{ep}` succeeds (and OSS is enabled), also `unpublish_ladder_from_staging` for each ladder. When `DELETE /admin/dramas/{slug}` succeeds, also `unpublish_drama_from_staging`. Eliminates the staging-side OSS orphan accumulation noted in CLAUDE.md.
- Add a one-shot migration helper `scripts/migrate_to_staging_prefix.py` that for each `status=ready` episode: re-uploads to staging prefix, rewrites the local m3u8. Used only if the operator has data from earlier OSS test runs they want to preserve. Not required when starting from scratch.

## Capabilities

### New Capabilities

- `oss-staging-prod-separation`: defines the staging/prod prefix layout, the `publish_ladder_to_prod` / `unpublish_*_from_prod` primitives, the prod-flavored m3u8 generation rule, and the staging-side cleanup tied to existing delete handlers.

### Modified Capabilities

- `oss-segment-hosting`: upload paths now include `staging/` prefix; locally-generated m3u8 references the staging absolute URL; `EpisodeInfo.initUrl` / `firstSegUrl` (when OSS enabled) reference the staging absolute URL. The `oss_public_base_url` constant's role narrows to "bucket-level base, for internal composition only."

## Impact

- **Code**: `app/oss_upload.py` (new constants, new helper functions), `app/publish.py` (`publish_ladder` paths change; new `publish_ladder_to_prod` / `unpublish_*` functions), `app/routers/api.py` (`_row_to_episode_info` uses `oss_staging_public_base_url`), `app/routers/admin.py` and the new drama-delete handler (added in `drama-as-entity`) call staging cleanup, `app/queue.py` worker invokes the new staging-prefix path.
- **Schema**: none. SQL is unchanged.
- **External contracts**: `EpisodeInfo` shape unchanged at the field level; `initUrl` / `firstSegUrl` values now contain `/staging/` in the path component when OSS is enabled. Step 6's business-server `EpisodeInfo` will contain `/prod/` instead.
- **OSS bucket**: gains a `Drama/staging/` and a `Drama/prod/` subtree. The legacy `Drama/{slug}/...` paths (without env prefix) are no longer written; pre-existing test objects under the old layout are orphaned and can be deleted manually or by the migration helper.
- **CORS**: unchanged (bucket-level CORS already allows GET from any origin, applies uniformly to both prefixes).
- **Downstream**: `business-server-sync` (step 6) consumes the new primitives — its sync handler calls `publish_ladder_to_prod` then ships the returned prod m3u8 to the business server.
- **No new external dependencies**.
